## 7. POTENTIAL ISSUES & MITIGATION

### Issue 1: Circular Dependencies
**Problem**: DeliveryService uses OrderRepository & PaymentRepository; OrderStatusService also uses both
**Impact**: 🟢 Low — No circular ref, just shared dependencies
**Solution**: That's fine, having shared repository dependencies won't create circular issues

### Issue 2: Shared Mapping Logic
**Problem**: OrderMappings.MapOrder() used by 3 different services
**Impact**: 🟡 Medium — Could lead to maintenance burden if Order entity changes
**Solution**: ✅ Create static `OrderMappings` class to centralize (cleaner than duplicating)

### Issue 3: GetAvailableTimeSlots() - Mock Data
**Problem**: This method returns hardcoded time slots (likely test data)
**Impact**: 🟡 Medium — Should eventually be replaced with real data source
**Solution**: 
- Option A: Extract to constants file for easier testing
- Option B: Create ITimeSlotProvider interface and inject real implementation
- **Recommendation**: Option A (simpler, sufficient for now)

### Issue 4: Service Responsibilities Overlap
**Problem**: Both OrderPlacementService & OrderStatusService work with Order entity
**Impact**: 🟢 Low — OrderPlacement handles NEW orders, OrderStatus handles existing order transitions (clear separation)
**Solution**: ✅ Clear responsibility boundary: Placement = Creation/Query, Status = Transitions/Mutations

### Issue 5: Configuration/DI Registration
**Problem**: Need to ensure all 4 services are properly registered in DI
**Impact**: 🟡 Medium — If forgotten, will cause runtime errors
**Solution**: Create extension method `AddOrderServiceLayer()` to register all 4 services at once

### Issue 6: Interface Contract Changes
**Problem**: IOrderWorkflowService still exists with all 21 methods
**Impact**: 🟡 Medium — Old interface might be used in controllers, causing confusion
**Solution**:
- Option A: Update controllers to use 4 new interfaces separately
- Option B: Keep IOrderWorkflowService, make it delegate to 4 services (adapter pattern)
- **Recommendation**: Option A (cleaner, forces proper separation)

### Issue 7: Cart Repository in PlaceOrderAsync
**Problem**: PlaceOrderAsync clears the cart after order creation
**Impact**: 🟢 Low — Works fine, CartService owns cart operations, OrderPlacementService just uses it
**Solution**: ✅ Consider if this should be a callback or use ICartService instead
